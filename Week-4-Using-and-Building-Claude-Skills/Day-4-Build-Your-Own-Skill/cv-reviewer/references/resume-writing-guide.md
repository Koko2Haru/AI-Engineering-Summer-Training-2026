# Resume Writing Guide

Structural and stylistic conventions for a strong modern resume, used both to inform scoring and to generate rewrite suggestions.

## Standard Section Order

1. **Contact Info** — name, phone, email, location (city/state is enough; full street address is rarely needed), LinkedIn/portfolio/GitHub if relevant
2. **Summary** (optional, 2-4 lines) — most valuable for career changers or senior candidates who need to frame scope upfront; often skippable for early-career candidates where space is better spent elsewhere
3. **Experience** — reverse chronological, most recent first
4. **Projects** — especially important for students, career changers, or when professional experience alone doesn't show the target skill set
5. **Education** — reverse chronological; can move above Experience for students/new grads
6. **Skills** — grouped by category (e.g., Languages, Frameworks, Tools) rather than one flat list
7. **Certifications / Achievements** (optional) — only if relevant and adds signal

Drop any section that doesn't add signal. A resume is a highlight reel, not a complete record.

## Length Norms

| Candidate Profile | Target Length |
|---|---|
| Student / entry-level (0-2 yrs) | 1 page |
| Mid-level (3-8 yrs) | 1 page, 2 only if content genuinely earns it |
| Senior / staff+ (8+ yrs) | 1-2 pages |
| Academic/research CV | Length norms differ entirely — publications, grants, and talks justify multiple pages |

Going over length to fit in weaker, less relevant content is worse than cutting to stay concise. Every line should be fighting for its place.

## The Bullet Point Formula (STAR / XYZ)

**STAR:** Situation, Task, Action, Result — useful for structuring the thinking behind a bullet, though the final bullet is usually just the Action + Result compressed into one line.

**XYZ formula (Google's version, good default target):**
> "Accomplished [X] as measured by [Y], by doing [Z]."

Example: "Reduced checkout abandonment by 18% (Y) by redesigning the payment form validation flow (Z) to surface errors inline instead of on submit (X, implicit)."

In practice, aim for: **[Strong action verb] + [what was done] + [quantified result/impact] + [how, if space allows]**.

- Weak: "Worked on improving the checkout flow."
- Better: "Redesigned the checkout form's validation flow."
- Strong: "Redesigned checkout form validation, reducing cart abandonment by 18% over one quarter."

## Tense and Voice

- Past roles: past tense throughout ("Led," "Built," "Reduced").
- Current role: present tense for ongoing responsibilities, past tense for completed achievements within that role, is acceptable — but be internally consistent.
- Never use first person ("I," "my") — bullets are implied first-person and drop the pronoun.
- Always active voice: "Automated the deployment pipeline," not "The deployment pipeline was automated by me."

## Contact Info Do's and Don'ts

**Do:**
- Use a professional email (firstname.lastname@ pattern or similar)
- Include city/state or "Remote" if relevant to role eligibility
- Include LinkedIn and GitHub/portfolio links when they add real signal (an active GitHub for engineers, a portfolio for designers)

**Don't:**
- Include a full street address (privacy risk, rarely used by recruiters)
- Include age, date of birth, marital status, or a photo unless standard practice in the candidate's country/industry
- Use an outdated or unprofessional-sounding email handle

## Summary Statements

When used, a summary should state: role/specialization, years of relevant experience, and 1-2 standout areas of impact or expertise. Avoid generic filler like "Hardworking team player seeking new opportunities" — this says nothing an employer couldn't assume by default and wastes prime real estate at the top of the page.

- Weak: "Motivated professional with strong communication skills looking for a challenging role."
- Strong: "Backend engineer with 5 years building high-throughput payment systems; led migration of a legacy monolith to microservices serving 2M+ daily transactions."

## Skills Section

Group by category rather than a flat alphabetical dump:

```
Languages: Python, TypeScript, Go
Frameworks: React, Django, FastAPI
Infrastructure: AWS, Docker, Kubernetes, Terraform
```

List skills the candidate can genuinely speak to in an interview — a long list of unfamiliar buzzwords is a liability, not an asset, once the interview starts.

## Tailoring to a Job Description

When a job description is available:
1. Extract the 8-12 most emphasized skills/requirements from the posting.
2. Check which are already present in the resume, in the posting's own terminology (not just a rough synonym).
3. Flag genuinely missing ones the candidate can truthfully add.
4. Reorder or emphasize existing bullets that map to the posting's top priorities — don't just append keywords disconnected from real experience.

## Formatting Consistency Checklist

- One font family throughout (a second, complementary font for headers is fine if used consistently)
- One bullet character/style throughout
- Consistent date format throughout
- Consistent bolding pattern (e.g., always bold job titles, never company names, or vice versa — pick one convention)
- Consistent spacing between sections and entries
