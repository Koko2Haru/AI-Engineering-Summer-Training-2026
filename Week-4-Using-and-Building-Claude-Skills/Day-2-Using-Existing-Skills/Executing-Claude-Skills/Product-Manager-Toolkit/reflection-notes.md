# Reflection Notes — `product-manager-toolkit` skill

> **Note:** These are my own personal notes, written by hand after the fact — **not** output generated by the skill itself. The skill only produced the RICE files linked at the bottom.

---

## How was it invoked

I called it explicitly: `/product-manager-toolkit` with the argument `RICE prioritization for CoinQuest v2.0 features`, plus a prompt listing the 8 candidate v2.0 features and my usage context (single user, ~1 session/day).

Invoking it didn't run any code by itself — it just loaded `SKILL.md` (the skill's instruction/reference doc) into context. Claude then had to read that doc, decide the RICE prioritizer script was the right tool for the ask, and drive it manually (create a CSV, run the Python script, interpret the output). So the skill itself is closer to **a reference manual + toolbox** than an automated pipeline — the actual work (choosing Reach/Impact/Confidence/Effort values, running the script, writing up the analysis) still happened in the conversation, not inside the skill.

## Input / Output

**Input:** a CSV I had Claude build (`coinquest-v2-features.csv`) — one row per feature with `reach, impact, confidence, effort, description` columns, following the schema documented in the skill's `SKILL.md`.

**Output:** running `scripts/rice_prioritizer.py` against that CSV produced:
- a ranked list of features by RICE score
- a portfolio analysis (quick wins vs. big bets, effort/impact distribution)
- an auto-generated quarterly roadmap based on a `--capacity` argument

Claude then wrote its own markdown summary on top of the raw script output (`RICE-Prioritization-CoinQuest-v2.md`) — that part is *not* the skill talking, that's Claude's interpretation layer.

## What did it create

Three files, all in this folder:

1. [`coinquest-v2-features.csv`](coinquest-v2-features.csv) — the RICE input table (name, reach, impact, confidence, effort, description) for all 8 backlog features.
2. [`rice_results.json`](rice_results.json) — raw JSON dump from the script: scored/sorted features + portfolio analysis + the (buggy) auto-roadmap.
3. [`RICE-Prioritization-CoinQuest-v2.md`](RICE-Prioritization-CoinQuest-v2.md) — the human-readable writeup: scored table, portfolio analysis, a callout on where RICE undersells the backlog, and a judgment-adjusted build order.

## What those files mean / contain

- The **CSV** is the only thing I'd call "my data" — it encodes the actual judgment calls (e.g. "Healing Passive = massive impact, medium effort"). Everything downstream is just math run on top of these numbers, so if the CSV values are off, everything else is off too.
- The **JSON** is machine-readable and meant for piping into other tools (the skill doc mentions Jira/dashboard integration) — not really meant to be read directly.
- The **markdown writeup** is the actual deliverable — it's where the RICE score gets *interpreted*, not just reported. This is also where the biggest value-add happened: the flat RICE ranking put "Reports & Comparison Tool" dead last and "Sound Design" 3rd, which is backwards from what actually matters for a finance tracker. The script has no way to catch that; only the write-up step does.

## Behaviour observed

- The skill is **entirely reference-driven** — `SKILL.md` is a table of contents pointing at `references/frameworks.md` (RICE formula + other frameworks) and `scripts/rice_prioritizer.py`. Nothing runs automatically; Claude has to choose to run the script.
- **RICE's `Reach` field assumes a multi-user product.** With just one user, a literal "users affected" count would flatten every feature to `Reach = 1` and break the ranking. Had to reinterpret Reach as "sessions/quarter the feature gets touched" to make the framework produce a meaningful ordering — the skill doesn't flag or handle this single-user case itself.
- **Hit a real bug**: `rice_prioritizer.py`'s text output uses emoji (📊, 🎯, 🚀) and printing it on Windows crashed with `UnicodeEncodeError` under the default `cp1252` console encoding. Fixed by setting `PYTHONIOENCODING=utf-8` before running. Worth remembering for next time on Windows.
- The **auto-generated roadmap** (capacity-based quarter bin-packing) is naive — with a low `--capacity` value it produced near-empty/overflowing quarters (e.g. "Q1: 0/4 used", "Q5: 8/4 used") because it doesn't split or defer effort gracefully. I ended up overriding it by hand in the writeup rather than trusting the script's roadmap section as-is.
- Overall the skill is useful as **scaffolding + a forcing function** (make you assign Reach/Impact/Confidence/Effort explicitly) but the score itself still needs a sanity check against actual product judgment before treating it as the final order.
