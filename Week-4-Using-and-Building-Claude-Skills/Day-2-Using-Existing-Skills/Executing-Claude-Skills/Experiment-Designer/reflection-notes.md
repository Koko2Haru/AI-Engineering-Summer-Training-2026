# Reflection Notes — `experiment-designer` skill

> **Note:** These are my own personal notes, written by hand after the fact — **not** output generated by the skill itself. The skill only produced the experiment-design files linked at the bottom.

---

## How was it invoked

I called it explicitly: `/experiment-designer`, with a prompt asking to test Roast mode vs Coach mode's effect on how often I log an expense in CoinQuest — baseline 3 entries/week, detect a 20% lift, give sample size and duration.

Same pattern as `product-manager-toolkit`: invoking the skill just loads its `SKILL.md` (workflow + pointers to `references/` and `scripts/`) into context. Nothing runs automatically. Claude had to read the workflow (hypothesis → metrics → sample size → stopping rule), decide the sample-size script applied, run it, and interpret/adapt the result — all of that is Claude driving the toolbox, not the skill acting on its own.

## Input / Output

**Input:** baseline rate (3 entries/week), target MDE (20% relative lift), plus context I gave in the prompt (single user, ~1 session/day) that isn't part of the skill's expected input shape at all.

**Output:** running `scripts/sample_size_calculator.py` against a reframed daily-proportion version of the metric (`3/7 ≈ 0.43` chance of logging on a given day) produced a required sample size, saved raw at [sample_size_raw_output.txt](sample_size_raw_output.txt). Claude then layered a second, hand-computed method (two-sample t-test on weekly counts) on top, since the tool's proportion-test assumptions didn't actually fit an N=1 count metric — that cross-check and the full write-up live in [experiment-design-roast-vs-coach.md](experiment-design-roast-vs-coach.md).

## What did it create

Two files, both in this `Output` folder:

1. [`sample_size_raw_output.txt`](sample_size_raw_output.txt) — raw stdout from `sample_size_calculator.py`: baseline/target rate, alpha, power, `n_per_group`, `n_total`, and estimated days.
2. [`experiment-design-roast-vs-coach.md`](experiment-design-roast-vs-coach.md) — the full experiment design: If/Then/Because hypothesis, primary/guardrail/secondary metrics, the crossover study design, the sample-size math (both methods), an MDE-vs-duration tradeoff table, a recommendation, a stopping rule, and N=1-specific validity threats.

## What those files mean / contain

- The **raw txt output** is the literal tool result treating "chance of logging on a given day" as a conversion rate — it says **533 days per arm (~3 years total)** are needed to detect the requested effect. Taken at face value this looks like a real, precise number, but it rests on an assumption (independent daily trials) that doesn't hold for a single person's repeated behavior — it's precise-looking, not necessarily *right*.
- The **markdown writeup** is where the actual thinking happened: recognizing the tool's assumption mismatch, redoing the math as a two-sample mean comparison on weekly counts (methodologically the correct framing for this design), and — critically — building the MDE-vs-duration table so the 20% MDE gets contextualized against what's actually achievable (turns out ~100% lift is the realistic floor for a few-month N=1 pilot). That table is the actual decision-useful artifact; the raw sample-size number by itself would have been misleading on its own.

## Behaviour observed

- Same as last time: **the skill is reference + tooling, not an autonomous pipeline.** `sample_size_calculator.py` only implements the two-proportion z-test — there's no built-in support for count/rate metrics or single-subject/crossover designs, even though the reference docs (`statistics-reference.md`) don't flag that limitation anywhere.
- **The tool silently accepted a mismatched input.** I fed it a reframed proportion (days-with-≥1-log) instead of the actual metric (entries/week), and it happily produced a confident-looking number (533 days) with zero warning that the underlying independence assumption was shaky for N=1 data. Nothing in the script or `SKILL.md` prompts a check for "is your unit of observation actually independent?" — that's a judgment call Claude/I had to bring from outside the tool, same as the single-user `Reach` issue in `product-manager-toolkit` last time.
- **Both the naive and the corrected method agreed on the headline conclusion** (this effect size is not detectable in a practical timeframe at N=1) even though they disagreed on the literal number (533 days vs. 131 weeks/917 days) — useful confirmation that the *qualitative* takeaway was robust even where the *point estimate* wasn't trustworthy.
- General pattern forming across both toolkit skills so far: they're built assuming a conventional multi-user product context (proportions, users-per-quarter), and applying them to a single-user personal project requires an explicit reframing step each time — the skill doesn't detect or flag the mismatch itself, that has to come from the person (or Claude) using it.
