# Reflection Notes — `md-slides` skill

> **Note:** These are my own personal notes, written by hand after the fact — **not** output generated by the skill itself. The skill only produced the three deck-pipeline files linked at the bottom.

---

## How was it invoked

I called it implicitly: pointing it at `SKILLS.md` (a 1,433-line developer reference doc under `Day-1-What-is-a-skill/Skills-in-Claude/`, documenting four *other* skills in depth) with my prompt and asking for it to become an HTML deck.

Worth flagging: I initially read the wrong thing. Loading `SKILLS.md` first made it look like *that file itself* was the skill's documentation — it happens to contain a whole chapter analyzing an `md-slides`-like skill. It isn't: it's the **input document** to convert, written about a similar-but-separate skill set. The *actual* installed skill lives at `.claude/skills/md-slides/` and is what actually ran. Same invocation pattern as the last two: `/md-slides` loads that real `SKILL.md` into context — nothing executes on its own, Claude has to run the three pipeline scripts by hand.

## Input / Output

**Input:** `SKILLS.md` — 1,433 lines, 1 H1 heading, **15** `---` HR lines, several huge sections (up to 284 source lines each) with nested H2/H3 headings, tables, and mermaid diagrams.

**Output:** three files from the pipeline (`slide_splitter.py` → `presenter_notes_parser.py` → `deck_html_renderer.py`), all saved to `Output/`:

1. [`slides.json`](Output/slides.json) — 16 raw slides, boundary mode auto-selected as `hr` (15 HRs ≥ the 3-HR threshold, so HR won over the single H1).
2. [`deck.json`](Output/deck.json) — same 16 slides with presenter notes extracted/stripped.
3. [`deck-skills-md.html`](Output/deck-skills-md.html) — the final single-file deck, 16 slides, 89,758 bytes.

## What did it create / what those files mean

- **`slides.json`** is the literal HR-split of the document: title = the doc's own headline, then one slide per numbered skill chapter (`1. product-manager-toolkit`, `2. experiment-designer`, `3. md-slides`, `4. caveman`), then one slide per closing appendix section (Comparison Matrix, Interaction Graph, etc.). The splitter printed a soft-warning for slides #5, #6, #7, #8, #13 exceeding the 40-source-line threshold — unsurprising, since each numbered-skill chapter is itself a ~280-line mini-document with 19 subsections.
- **`deck.json`** adds a `notes` field per slide. Coverage: **1/16 slides (6.2%)** — see the bug below, because that 1 "note" isn't a real presenter note.
- **`deck-skills-md.html`** is the actual deliverable: opens in any browser, arrow keys / Space / PgDn move forward, `P` toggles presenter split-view, `Cmd+P` prints one slide per page to PDF.

## Behaviour observed

- **Same reference-driven pattern as the other two skills** — `SKILL.md` describes a 3-script pipeline; nothing runs until Claude reads it and drives the scripts by hand.
- **Boundary auto-detection worked as documented**: 15 HRs cleared the `≥3` threshold, so it split on `---` rather than the single H1. Correct call for this doc.
- **Hit a real bug — false-positive presenter notes.** `presenter_notes_parser.py` regex-matches literal `<!-- notes: ... -->` text anywhere in a slide's body, with no awareness of Markdown code-span backticks. `SKILLS.md`'s own "3. md-slides" chapter *documents* the notes syntax using inline code like `` `<!-- notes: ... -->` `` (explaining the feature, not using it) — the parser matched that literal text as if it were a real presenter note, stripped the string `...` out of slide 7's visible body, and reported it as "notes." A skill that reads about itself trips its own regex.
- **Bigger issue — silent rendering degradation.** `deck_html_renderer.py` imports a markdown parser from a *sibling skill*, `md-document`, and design tokens from another sibling, `design-system` — neither exists in this install (`.claude/skills/` only has `md-slides`, `product-manager-toolkit`, `experiment-designer`, `caveman`, plus a few others; no `md-document` or `design-system`). Both imports fail, get caught by a bare `except ImportError`, and silently fall back: `_mp = None` means slide bodies render as one big **escaped `<pre>` block** instead of parsed headings/lists/tables/callouts. Confirmed by counting output tags: **16 `<pre>` blocks (one per slide), 0 `<h2>`, 0 `<table>`** — every table and mermaid diagram in the source doc is now a wall of raw monospace markdown text inside the slide, not a rendered element. The deck still opens and navigates fine; it just doesn't look like a "presentation" for any slide with real Markdown structure. Nothing in `SKILL.md` or the script's `--help` mentions this dependency or that it degrades this silently — I only found it by grepping the output HTML.
- **Net result:** the deck technically satisfies every hard rule (≥3 HR boundaries, 16 slides ≠ 1, ≥100 lines, no `--strict-notes` requested) and rendered successfully, but two things need a human's judgment call the skill won't surface on its own: (1) the false-positive note on slide 7 should be manually cleared, and (2) without the `md-document`/`design-system` sibling skills installed, this is really a *raw-markdown paginator*, not a formatted deck — fine for personal review, not what I'd hand to someone expecting Keynote-quality slides.
